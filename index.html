Git Basics

‚è≥ 1. Git - What is it? Why is it needed?

Git is a Version Control System (VCS). Think of it as a "Time Machine" for your code. It tracks every change made to a project, who made it, and when.

Why is it needed?

Backup and History If you delete a file by mistake or break your code, you can jump back to a version from yesterday or last year.
Collaboration Multiple people can work on the exact same file at the same time without overwriting each other's work.
Experimentation You can create a "branch" to try a crazy idea without touching the stable version of your app.

üèóÔ∏è 2. Git Basics: The Core Concepts

To understand Git, you should understand how your files change and move from one phase to another. Your code moves through three distinct phases:

The Workflow Areas

Working Tree (Directory) The actual folder on your computer where you are editing files.
Staging Area (Index) A "pre-commit" area. It‚Äôs like a shopping cart where you put the files you want to save in the next snapshot.
Local Repository (.git) Where Git stores the compressed history of all your commits.

üîë Key Terms

Commit

A "snapshot" of your project at a specific point in time. Each commit has a unique ID (SHA).

Branch

A parallel version of your repository. It diverges from the main line to allow for isolated work.

HEAD

A pointer that tells Git which branch or commit you are currently looking at.

Tag

A permanent "bookmark" for a specific commit, usually used to mark version releases (e.g., v1.0).

2 Branching

What is a Branch?

A branch in Git is like a separate line of development‚Äîa copy of your project where you can make changes without affecting the main code. Think of it as a parallel universe for your code:

  • You start from main (the stable version).
  • Create a branch for a feature or fix.
  • Work independently, commit changes, and later merge back into main when ready.
  • Branches let multiple developers work on different features at the same time, keep the main branch clean, and avoid code freezes.

Why Branching Matters

Branching is Git‚Äôs superpower for collaboration and parallel development. It lets you work on multiple features without code freezes, enables teams worldwide to collaborate via GitHub, and keeps the main branch stable while experimenting on others.

3 Branching Strategies and Best Practices

Git Flow

A structured model using main for production, develop for integration, and feature/ branches for tasks.

GitHub Flow

A simpler model where everything in main is always deployable, and all work happens in short-lived feature branches.

Team Best Practices

01 Keep it simple
  • Use feature branches for all new work.
  • Merge into main via pull requests.
  • Keep main clean, tested, and up-to-date.
02 Branch naming

Use clear conventions:

feature/feature-name
bugfix/description
users/username/workitem
03 Pull requests
  • Always review before merging.
  • Include build/details for easy testing.
  • Aim for at least two reviewers.
04 Release branches
  • Create release/x.y for milestones.
  • Fix bugs in release branches.
  • Port changes to main via cherry-pick.
05 Policies for main
  • Require PRs for all merges.
  • Require successful automated builds.
  • Use auto-added reviewers.
06 Tags vs branches

Prefer branches for releases; tags are optional markers for specific commit points.

4 Merging

Think of Git Merging as the moment where different streams of work finally come together to form a finished product.

If you‚Äôve been working on a "Feature" branch while your teammate was fixing bugs on the "Main" branch, merging is the process that combines those separate histories into one.

Why is Merging Required?

Isolation

Build features without breaking the working version of the app.

Collaboration

Enables multiple developers to work on the same project simultaneously.

Integration

Merging brings new code into the project's source of truth.

The Two Main Merge Techniques

Fast-Forward Merge

This is the simplest type of merge. It happens when the main branch hasn't moved since you created your feature branch.

When it happens: No new commits have been made on the base branch.

The Result: A perfectly linear project history.

How to Perform a Merge (The Workflow)

1. Switch to receiving branch
git checkout main
2. Pull latest changes
git pull origin main
3. Merge the feature branch
git merge feature-login
4. Push the result
git push origin main

5 Rollback: Fixing Mistakes

Git Revert

(Safe Way)

Creates a new commit that undoes a previous one. Doesn't delete history.

Git Rebase

(Clean History)

Moves or combines a sequence of commits to a new base commit.

Git Reset

(Nuclear Way)

Moves the branch pointer backward. Warning: Never use on shared commits!
