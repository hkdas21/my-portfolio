2 Branching

What is a Branch?

A branch in Git is like a separate line of development‚Äîa copy of your project where you can make changes without affecting the main code. Think of it as a parallel universe for your code:

  • You start from main (the stable version).
  • Create a branch for a feature or fix.
  • Work independently, commit changes, and later merge back into main when ready.
  • Branches let multiple developers work on different features at the same time, keep the main branch clean, and avoid code freezes.

Why Branching Matters

Branching is Git‚Äôs superpower for collaboration and parallel development. It lets you work on multiple features without code freezes, enables teams worldwide to collaborate via GitHub, and keeps the main branch stable while experimenting on others.

3 Branching Strategies and Best Practices

Adopt a Git Branching Strategy

A clear branching strategy keeps collaboration smooth and prevents history from becoming a "spaghetti mess." Follow these industry principles:

01

Keep it simple

  • Use feature branches for all new work.
  • Merge into main via pull requests.
  • Keep main clean, tested, and up-to-date.
02

Branch naming

Use clear conventions:

feature/feature-name
bugfix/description
users/username/workitem
03

Pull requests

  • Always review before merging.
  • Include build/details for easy testing.
  • Aim for at least two reviewers.
04

Release branches

  • Create release/x.y for milestones.
  • Fix bugs in release branches.
  • Port changes back to main via cherry-pick.
05

Policies for main

  • Require PRs for all merges.
  • Require successful automated builds.
  • Use auto-added reviewers.
06

Tags vs branches

Prefer branches for releases; tags are optional markers for specific commit points.
